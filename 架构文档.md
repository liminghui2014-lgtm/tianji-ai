## 二、关键设计决策

### 2.1 为什么用 Node.js 排盘而不是 Python？

iztro 是目前唯一开源的完整紫微斗数排盘库，只支持 JavaScript/Node.js。Python 端通过 `subprocess` 调用 `chart_calculator.js`，用 JSON 通信。不是最优解，但是最可用的解。

**瓶颈**: Node.js 冷启动约 1-2 秒。Streamlit Cloud 上每次部署需要重新 `npm install`。

### 2.2 真太阳时为什么重要？

北京时间是东八区（120°E）的标准时间。但昆明（102.7°E）在 120°E 以西 17.3 度，实际太阳时比北京晚约 69 分钟。不用真太阳时校正，昆明出生的人的时辰会错一个。

算法: 时差方程(EoT) + 经度偏移 → 真太阳时 → 时辰索引。

### 2.3 格局检测为什么不用 AI？

LLM 的优势是非结构化推理，劣势是确定性规则。紫微斗数的格局有明确定义——"七杀破军贪狼都在命宫三方四正=杀破狼"是硬规则。用代码规则引擎: 100% 准确、零幻觉、零延迟。用 AI: 可能漏检、可能编造格局名。

AI 只负责最后一环——解读。格局检测保持确定性。

### 2.4 Prompt 设计的核心原则

1. **证据锚定**: 每条判断必须引用具体星曜+宫位。不说"你适合创业"，要说"命宫七杀在寅，三方见破军贪狼，杀破狼格局——开创型人格"。
2. **分层解读**: 本命（出厂设置）→ 大限（十年运势）→ 流年（当年变化）。三层不混讲。
3. **话不说死**: "倾向"而非"注定"，"容易"而非"必然"。紫微不是宿命论。
4. **温度把控**: 倪海夏风格的精髓是"诊断人生，开方子"——有医者的慈悲，不是算命先生的恐吓。
5. **天纪语料加持**: RAG 检索倪师原话注入 prompt，让 AI 输出带有倪师的语言指纹。

## 三、RAG 架构 (v2 — Hybrid)

```
天纪字幕.txt (87万字)
  │
  ▼
语义切块 (按空行/章节边界, 每块800-2000字)
  │
  ▼
┌──────────────────────────────────────────────┐
│         Hybrid RAG 检索管道                    │
│                                               │
│  用户命盘 → 提取星曜/宫位/四化 → 构建查询       │
│         │                                     │
│         ├──→ BM25 通道 (权重 0.6)              │
│         │    jieba分词 + 紫微术语保护           │
│         │    命中星曜名/宫位名/格局名硬匹配      │
│         │                                     │
│         └──→ Embedding 通道 (权重 0.4)          │
│              paraphrase-multilingual-MiniLM    │
│              捕获"适不适合创业"等语义意图        │
│                                               │
│         混合打分 → Top 20 → 截取6000 tokens     │
└──────────────────────────────────────────────┘
  │
  ▼
注入 prompt → DeepSeek 解读
```

**v1 → v2 升级**:
- 切块粒度: 按集(粗) → 按段落边界(细, 每块800-2000字)
- 检索方式: 关键词命中 → BM25 + Embedding 混合
- 语义能力: 无 → Embedding 捕获非结构化查询意图
- 术语保护: 无 → jieba 分词时保护"杀破狼"等复合词不被拆散
- 持久化: 每次重算 → 索引保存到 `.rag_index/`, 修改语料后重新构建

## 四、数据流

```
┌──────────┐
│  users   │ 姓名/性别/城市/生辰/四柱/出生经纬度
├──────────┤
│  charts  │ 完整命盘JSON(十二宫+星曜+四化+五行局) + 格局列表 + AI解读全文
├──────────┤
│  chats   │ 对话记录: 用户问题 + AI回复 (关联 user_id + chart_id)
├──────────┤
│payments  │ 付费记录 (预留)
├──────────┤
│analytics │ 事件埋点 (预留)
└──────────┘
全部存在 SQLite (tianji.db)，一张表一个文件。
```

## 五、Streamlit Cloud 部署

- 平台: Streamlit Cloud (免费 tier)
- 运行时: Python 3.11 + Node.js 18
- 启动: `streamlit run app.py`
- 冷启动: ~30-60 秒 (含 npm install)
- 内存限制: 1GB (RAG 索引需控制在 300MB 内)

## 六、v0.2 架构优化 (2026-05-12)

| 优化项 | 原方案 | 新方案 |
|--------|--------|--------|
| **模型温度** | Temperature 0.8/0.85 | Temperature 0.3 (平衡命理严谨+语言风格) |
| **幻觉熔断** | 无 | System Prompt 顶层: 缺大限流年数据时强制声明"仅基于本命盘推演", 严禁捏造年份运势 |
| **上下文压缩** | 每次对话传完整命盘(4000+ tokens) | 首轮: 完整命盘; 后续: 仅传核心格局+命宫关键星曜+最近3轮对话摘要 |
| **免费次数校验** | `st.session_state` (客户端, 可绕过) | SQLite `chats` 表计数 → `consume_free_chat()` 服务端强校验 |
| **付费墙** | 刷新页面即重置 | 服务端计数器, 用完为止, 不可绕过 |

## 七、已知技术债

1. **Node.js 子进程开销**: 每次解读 `subprocess.run` 调 Node（冷启动1-2秒）。方案: 封装为独立微服务(Express)通过 HTTP 通信
2. ~~**RAG 关键词匹配粗糙**~~ → ✅ v2: BM25(0.6) + Embedding(0.4) + jieba 分词 + 语义切块
3. **无真实大限流年数据**: iztro `getHoroscope()` 输出不稳定。当前 AI 声明"仅基于本命盘推演"。方案: 修复流年计算或接入其他排盘引擎
4. **Streamlit Cloud 数据非持久化**: 每次部署 SQLite 被清空。方案: 迁移到 Supabase (PostgreSQL)
5. **首次索引构建**: sentence-transformers 模型下载约 470MB + 编码全语料约 3-5 分钟 (CPU)。后续从 `.rag_index/` 加载仅需秒级
